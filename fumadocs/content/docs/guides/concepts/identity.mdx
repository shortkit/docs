---
title: "Identity"
description: "How ShortKit tracks anonymous and identified users across sessions and devices, and how identity resolution merges their history."
---

ShortKit assigns every device an anonymous ID on first launch. When your user signs in, you promote them to an identified user and ShortKit merges their anonymous history into the identified profile. Every analytics metric, view record, and event is attributed through this identity model — the `viewer_id` dimension you filter on in [analytics](/docs/guides/analytics/overview) is the ID described on this page.

## Anonymous by default

On first initialization, the SDK generates a random UUID and persists it in device storage (browser local storage on Web). This anonymous ID is attached to every API request and analytics event automatically via the `X-User-Id` request header. No setup is required.

The anonymous ID survives app restarts and page reloads. It is only replaced when you call `clearUserId()`, which discards it and generates a fresh one.

## Setting an identified user

Call `setUserId` when the user signs in. The SDK switches all subsequent requests and events to this ID and triggers [identity resolution](#how-identity-resolution-works) on the backend.

<Tabs groupId="platform" persist items={['iOS','Android','Flutter','React Native','Web']}>
  <Tab value="iOS">
    ```swift
    shortKit.setUserId("user_abc123")
    ```
  </Tab>
  <Tab value="Android">
    ```kotlin
    shortKit.setUserId("user_abc123")
    ```
  </Tab>
  <Tab value="Flutter">
    ```dart
    final controller = ShortKit.of(context);
    await controller.setUserId('user_abc123');
    ```
  </Tab>
  <Tab value="React Native">
    ```tsx
    import { useShortKit } from '@shortkitsdk/react-native';

    function useSignIn() {
      const { setUserId } = useShortKit();
      return (id: string) => setUserId(id);
    }
    ```

    Alternatively, drive identity declaratively: set the `userId` prop on `ShortKitProvider` and the SDK calls `setUserId` for you whenever the prop changes. Clearing the prop reverts to anonymous mode.
  </Tab>
  <Tab value="Web">
    ```js
    sk.setUserId('user_abc123');
    ```
  </Tab>
</Tabs>

If the user is already signed in when your app launches, pass the user ID at initialization instead of calling `setUserId` afterwards:

<Callout type="info" title="Use your own user IDs">
The user ID is any stable string from your own system — a database primary key, an auth provider subject, or a hashed account identifier. ShortKit treats it as opaque. Avoid raw email addresses or other direct personal identifiers.
</Callout>

## Clearing identity

Call `clearUserId` on sign-out. The SDK reverts to anonymous mode with a **fresh** UUID — the previous anonymous ID is discarded, so activity after sign-out is never linked back to the signed-in user.

  <Tab value="iOS">
    ```swift
    shortKit.clearUserId()
    ```
  </Tab>
  <Tab value="Android">
    ```kotlin
    shortKit.clearUserId()
    ```

  <Tab value="Flutter">

## How identity resolution works

When `setUserId` is called, the SDK sends a `POST /v1/identity/resolve` request (authenticated with your publishable key) carrying both the device's anonymous ID and the identified user ID:

```json title="Request body"
{
  "anonymousId": "a1b2c3d4-5e6f-4a7b-8c9d-0e1f2a3b4c5d",
  "userId": "user_abc123"
}
```

The backend then:

1. **Finds or creates** the identified user profile for `userId` in your organization.
2. **Links the device** by appending the anonymous ID to the identified profile's linked device IDs (deduplicated — resolving the same pair twice is a no-op).
3. **Deletes the anonymous profile.** All events and views previously attributed to the anonymous ID become queryable under the identified user.

```json title="Response"
{
  "data": { "status": "ok" },
  "meta": { "request_id": "req_01h..." }
}
```

Because the same `userId` can be resolved from multiple devices, an identified profile accumulates one linked device ID per device the user signs in on. This is how ShortKit unifies a user's history across their phone, tablet, and browser.

The SDK calls this endpoint for you. It is part of the SDK runtime (publishable-key) surface, not the public REST API, so it has no endpoint reference page.

## Identity on analytics events

Every analytics event the SDK emits carries the current user ID — anonymous or identified — in the `X-User-Id` request header. You never attach user IDs to events manually; the SDK handles it at the network layer. The same header attributes feed requests, so personalized ranking and watch-history deduplication follow the user across sign-in.

To attach additional user attributes to events (plan tier, cohort, A/B variant), use [custom dimensions](/docs/guides/analytics/custom-data) — they complement the user ID rather than replacing it.
