title: "Webhooks"
description: "Callback event reference for uploads, imports, clips, and live stream recordings — payload shapes, delivery, and retry semantics."
---

ShortKit can push processing events to your backend instead of making you poll. There are two delivery mechanisms:

- **Per-request callbacks** — pass a `callbackUrl` in the request body when you [upload](/docs/api/content/upload-content), [import](/docs/api/content/import-content), or [create clips](/docs/api/live-streams/create-clips). ShortKit POSTs to that URL as the resulting content moves through the pipeline.
- **Organization webhook** — set a **Live VOD Webhook URL** in your organization settings in the ShortKit dashboard. ShortKit POSTs a `content.playback_id_updated` event there whenever an ended live stream's recording becomes available.

Your endpoint must be reachable over HTTPS and respond with a `2xx` status code to acknowledge receipt. ShortKit waits up to 10 seconds for a response before treating the attempt as failed.

## Event types

| Event | Trigger | Delivered via |
|-------|---------|---------------|
| `content.imported` (`status: "processing"`) | File received, transcoding started | `callbackUrl` |
| `content.imported` (`status: "ready"`) | Transcoding complete, all quality levels available | `callbackUrl` |
| `content.import_failed` (`status: "error"`) | Processing failed | `callbackUrl` |
| `content.imported` (carousel) | All carousel images validated | `callbackUrl` |
| `content.playback_id_updated` | Ended live stream's recording is available; playback ID swapped | Organization webhook |

## Video events

ShortKit delivers up to three callbacks per video, whether it arrived by direct upload, URL import, or as a clip from a live stream. The `source` field tells you which: `"upload"`, `"url"`, or `"clip"`.

### `content.imported` with `status: "processing"`

Sent when the file has been received and transcoding has begun.

```json
{
  "event": "content.imported",
  "contentId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "externalId": null,
  "status": "processing",
  "playbackId": "rJ4OE02Z0Z4IY01Eo4Vgup5OilG8xZB9",
  "title": "Product Launch Teaser",
  "source": "upload",
  "timestamp": "2026-03-10T14:30:00.000Z"
}
```

If a `playbackId` is present, store it alongside your content record — the video is already streamable at this point, though transcoding is not yet complete. Viewers of longer videos may experience buffering near the end until encoding finishes.

<Callout type="info" title="playbackId can be null here">
`playbackId` may be `null` in the `processing` callback if processing is still in its earliest stage. The `ready` callback always includes a non-null `playbackId`.
</Callout>

### `content.imported` with `status: "ready"`

Sent when transcoding completes and the video is fully available at all quality levels. The payload shape is identical to the `processing` callback, with `status` set to `"ready"` and `playbackId` always present.

### `content.import_failed` with `status: "error"`

Sent if processing fails. The payload includes an `error` field with a human-readable description of the failure. No `playbackId` is included.

```json
{
  "event": "content.import_failed",
  "contentId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "externalId": null,
  "status": "error",
  "error": "Video processing failed",
  "title": "Product Launch Teaser",
  "source": "upload",
  "timestamp": "2026-03-10T14:35:00.000Z"
}
```

### Field reference

| Field | Type | Description |
|-------|------|-------------|
| `event` | string | `"content.imported"` or `"content.import_failed"` |
| `contentId` | string | The content item's UUID |
| `externalId` | string \| null | The idempotency key you provided on the original request, or `null` |
| `status` | string | `"processing"`, `"ready"`, or `"error"` |
| `playbackId` | string \| null | Playback ID used to resolve streaming and thumbnail URLs; omitted on failure events |
| `error` | string | Failure description; only on `content.import_failed` |
| `title` | string | The content item's title |
| `source` | string | `"upload"` (direct upload), `"url"` (URL import), or `"clip"` (live stream clip) |
| `timestamp` | string | ISO 8601 timestamp of the event |

## Carousel events

Carousel uploads deliver a single callback once all images have been validated. The `status` is `"ready"` if at least one image processed successfully, or `"errored"` if all images failed. Only successfully processed images appear in the `images` array.

```json
{
  "event": "content.imported",
  "contentId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "externalId": null,
  "contentType": "carousel",
  "status": "ready",
  "title": "Top 5 Weekend Destinations",
  "images": [
    { "position": 0, "url": "https://..." },
    { "position": 1, "url": "https://..." },
    { "position": 2, "url": "https://..." }
  ],
  "timestamp": "2026-03-10T14:30:00.000Z"
}
```

<Callout type="info" title="Carousels created via import are already ready">
URL imports only send callbacks for **video** items. Carousels submitted through the import endpoint are processed synchronously and are already `ready` in the initial import response — no callback is sent for them.
</Callout>

## Live stream recording events

### `content.playback_id_updated`

When a live stream ends and its recording becomes available, ShortKit swaps the stream's playback ID from the live broadcast to the on-demand recording. If your organization has a Live VOD Webhook URL configured, ShortKit POSTs this event so your backend can react to the change — for example, updating a stored playback ID or publishing the recording.

```json
{
  "event": "content.playback_id_updated",
  "content_id": "5a7b8c9d-e0f1-4234-5678-90abcdef1234",
  "org_id": "9f8e7d6c-5b4a-3210-fedc-ba9876543210",
  "new_playback_id": "DEFuvw456NewRecordingPlaybackId",
  "previous_playback_id": "ABCxyz123LiveBroadcastPlaybackId",
  "timestamp": 1749571200
}
```

| Field | Type | Description |
|-------|------|-------------|
| `event` | string | Always `"content.playback_id_updated"` |
| `content_id` | string | The live stream's UUID — the same `id` returned by [Create live stream](/docs/api/live-streams/create-live-stream) |
| `org_id` | string | Your organization's UUID |
| `new_playback_id` | string | Playback ID of the on-demand recording |
| `previous_playback_id` | string | Playback ID of the live broadcast being replaced |
| `timestamp` | integer | Unix epoch seconds |

<Callout type="warn" title="Different field naming">
This event uses snake_case field names and a numeric timestamp, unlike the camelCase `content.imported` family. Match on the `event` field, not on payload shape.
</Callout>

This event is configured per organization, not per request: set the webhook URL in the ShortKit dashboard under your organization settings. See [Live streams](/docs/guides/content/live-streams) for the full live-to-recording lifecycle.

## Retry behavior

Each callback is attempted up to **3 times**. If your endpoint returns a non-2xx status code or is unreachable, ShortKit retries after 1 second, then again after 5 seconds. If all three attempts fail, the callback is abandoned.

Callbacks are fire-and-forget — a failed callback never affects processing. The video continues transcoding, the clip is still created, and the recording swap still happens whether or not your endpoint acknowledged the event.

## Handling missed callbacks

If your endpoint was down and you missed a callback, retrieve the current state by polling the content item directly:

```bash title="Check current status"
curl https://api.shortkit.dev/v1/content/a1b2c3d4-e5f6-7890-abcd-ef1234567890 \
  -H "Authorization: Bearer sk_live_your_secret_key"
```

The response includes the item's `uploadStatus` and playback ID. See [Get content](/docs/api/content/get-content).

## Securing your endpoint

Callback requests are not signed. Treat incoming events as hints rather than authoritative state:

- Include a hard-to-guess token in the URL you register (for example `https://your-backend.com/webhooks/shortkit/3f9a1c70d2e84b56`) and reject requests to other paths.
- On receipt, confirm the reported state with `GET /v1/content/{id}` before acting on it. This also makes your handler idempotent if a retry delivers the same event twice.
- Respond `2xx` quickly and do heavy work asynchronously — ShortKit times out after 10 seconds and counts that as a failed attempt.

## Where to register callback URLs

| Flow | How to register |
|------|-----------------|
| Direct upload | `callbackUrl` body field on [`POST /v1/content/upload`](/docs/api/content/upload-content) |
| URL import | `callbackUrl` body field on [`POST /v1/content/import`](/docs/api/content/import-content) |
| Live stream clips | `callbackUrl` body field on [`POST /v1/live-streams/{id}/clips`](/docs/api/live-streams/create-clips) |
| Live recording playback ID swap | Live VOD Webhook URL in organization settings (ShortKit dashboard) |

For the task-oriented walkthrough of getting videos in and tracking them to `ready`, see [Upload videos](/docs/guides/content/upload-videos).
