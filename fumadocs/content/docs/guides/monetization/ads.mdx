ShortKit monetization has two halves:

1. **Server-side placement.** You set an ad configuration for your organization via the [Ad Configuration API](/docs/api/ad-configuration/get-ad-configuration). When enabled, the server interleaves `ad_slot` items between content in every feed response — you never compute ad positions client-side.
2. **Client-side rendering.** The SDK hands each ad slot to a `ShortKitAdProvider` you supply at initialization. The provider loads and displays the ad using your ad stack (Google IMA, Google Ad Manager, AppLovin, or a custom ad server).

<Callout type="info" title="Platform support">
Ad provider rendering is available on iOS and Android. React Native, Flutter, and Web do not yet support ad slots — the feed serves them content only. Sponsored items (see below) are served as ordinary video items and play on every platform.
</Callout>

## Configure ad placement

The ad configuration is a single per-organization object managed with a secret key. Three fields control where ads appear:

| Field | Default | Meaning |
|---|---|---|
| `enabled` | `false` | Master switch. When `false`, no ad slots are injected. You can still update the other fields while ads are disabled; the changes take effect when you re-enable. |
| `firstAdOffset` | `3` | Position of the first ad, counted in content items. `3` means the first ad appears after the third item. |
| `frequency` | `5` | Content items between consecutive ads after the first. |
| `maxPerPage` | `3` | Cap on ad slots per feed page. |

```
Content 1
Content 2
Content 3
  — Ad slot 1 (firstAdOffset: 3) —
Content 4
Content 5
Content 6
Content 7
Content 8
  — Ad slot 2 (frequency: 5) —
Content 9
...
```

Skip behavior is also set here and passed through to the SDK in each slot:

| Field | Default | Meaning |
|---|---|---|
| `skipPolicy` | `"skippable_after"` | One of `"non_skippable"`, `"skippable_after"`, `"always_skippable"`. |
| `skipDelaySeconds` | `5` | Seconds before the skip button appears. Only used with `"skippable_after"`. |
| `defaultTargeting` | `{}` | Key-value pairs copied into every slot's `targeting` field for your ad server. |

Updates are sparse — only the fields you send change:

## Ad formats

The configuration supports three ad sources. They are mutually exclusive per slot; sponsored takes precedence, then native, then VAST.

### VAST video ads

Set `adTagUrl` to a VAST/VMAP ad tag URL (and optionally `adUnitId` for your ad platform). Both are passed through in each ad slot so the client-side ad player — typically a Google IMA integration — can request creatives from your ad server. The ad provider renders its own player UI inside the slot.

### Native ads

Set `nativeEnabled: true` to serve slots with `adFormat: "native"`. Native ads play through ShortKit's own player pool, so they scroll, preload, and loop like regular feed items — your overlay supplies the headline, sponsor logo, CTA, and policy chrome. The `nativeFormatId`, `nativeAdUnitId`, and `native*Key` fields map your ad platform's native response keys to the asset names the SDK expects (`videoUrl`, `headline`, `ctaText`, `sponsorName`, `sponsorLogo`).

### Sponsored items

Set `sponsoredStreamingUrl` (with optional `sponsoredHeadline` and `sponsoredCta`) to run a self-served house ad. Instead of an ad slot, the server emits a regular video item at each ad position with `customMetadata.adFormat = "sponsored"` plus the headline and CTA. Because it is an ordinary content item, it plays on every platform with no ad provider required.

## Ad slots in the feed

When ads are enabled, feed responses contain `ad_slot` items between content items:

```json
{
  "type": "ad_slot",
  "id": "ad_slot_p0_3_0",
  "position": 3,
  "adTagUrl": "https://ads.example.com/vast?placement=shortkit",
  "adUnitId": null,
  "targeting": { "content_type": "news" },
  "skipPolicy": "skippable_after",
  "skipDelaySeconds": 3,
  "adFormat": null,
  "nativeFormatId": null,
  "nativeAssetKeys": null
}
```

The SDK decodes each slot into an `AdSlot` value and hands it to your ad provider. If no ad provider was passed at SDK initialization, ad slots are filtered out of the feed and viewers see content only — it is always safe to enable ads server-side before the client integration ships.

## Render ads in the SDK

Pass an implementation of `ShortKitAdProvider` when you initialize the SDK. The SDK calls `loadAd` when a slot enters the prefetch window (about 3 items ahead of the current position), `displayAd` when the slot scrolls on screen, and `dismissAd` when the user swipes away or the ad finishes. If a load fails (no fill, timeout), the SDK collapses the slot and the feed scrolls past it.

<Tab value="iOS">
```swift title="MyAdProvider.swift"
import ShortKitSDK

final class MyAdProvider: ShortKitAdProvider {
    func loadAd(for slot: AdSlot, completion: @escaping (Result<PreparedAd, AdError>) -> Void) {
        adServer.requestAd(tagUrl: slot.adTagUrl, targeting: slot.targeting) { ad, error in
            if let ad {
                completion(.success(PreparedAd(
                    slotId: slot.id,
                    skipPolicy: slot.skipPolicy,
                    providerData: ad
                )))
            } else {
                completion(.failure(.noFill))
            }
        }
    }

    func displayAd(
        _ ad: PreparedAd,
        in container: UIView,
        presentingViewController: UIViewController,
        delegate: ShortKitAdDelegate
    ) {
        // Attach the ad view to `container` and start playback.
        // Report lifecycle events to `delegate`.
    }

    func dismissAd(_ ad: PreparedAd) {
        // Stop playback and release the ad.
    }
}

let shortKit = ShortKit(
    apiKey: "pk_live_your_publishable_key",
    adProvider: MyAdProvider()
)
```

Your provider reports playback events through the `ShortKitAdDelegate` passed to `displayAd`. The SDK uses these for analytics and to advance the feed:

| Method | Call when |
|---|---|
| `adDidStart(_:)` | Ad playback begins. |
| `adDidReachQuartile(_:quartile:)` | Playback crosses 25% (`.first`), 50% (`.midpoint`), or 75% (`.third`). |
| `adDidComplete(_:)` | Ad finishes. |
| `adWasSkipped(_:watchTime:)` | User taps skip; pass watch time in seconds. |
| `adDidFail(_:error:)` | Playback fails after display. |
| `adDidReceiveClick(_:)` | User taps the ad. |

`AdError` cases: `.noFill`, `.timeout`, `.networkError(String)`, `.providerError(String)`. `PreparedAd.skipPolicy` is a typed enum: `.nonSkippable`, `.skippableAfter(seconds:)`, or `.alwaysSkippable` — enforce it in your ad UI.
</Tab>
<Tab value="Android">
```kotlin title="MyAdProvider.kt"
import android.view.ViewGroup
import com.shortkit.sdk.ShortKit
import com.shortkit.sdk.ads.PreparedAd
import com.shortkit.sdk.ads.ShortKitAdProvider
import com.shortkit.sdk.model.AdSlot

class MyAdProvider : ShortKitAdProvider {
    override suspend fun loadAd(adSlot: AdSlot, container: ViewGroup?): PreparedAd? {
        val ad = adServer.requestAd(adSlot.adTagUrl, adSlot.targeting)
            ?: return null  // null = no fill; the SDK collapses the slot
        return PreparedAd(
            slotId = adSlot.id,
            skipPolicy = adSlot.skipPolicy,
            providerData = ad,
        )
    }

    override fun displayAd(preparedAd: PreparedAd, container: ViewGroup) {
        // Attach the ad view to `container` and start playback.
    }

    override fun dismissAd(preparedAd: PreparedAd) {
        // Stop playback and release the ad.
    }
}

val shortKit = ShortKit(
    context = applicationContext,
    apiKey = "pk_live_your_publishable_key",
    adProvider = MyAdProvider(),
)
```

`loadAd` is a suspend function: return `null` for no fill, or throw to report a provider error — both collapse the slot. `PreparedAd.skipPolicy` is a sealed class: `SkipPolicy.NonSkippable`, `SkipPolicy.SkippableAfter(seconds)`, or `SkipPolicy.AlwaysSkippable` — enforce it in your ad UI.
</Tab>
</Tabs>

For native ads, set `PreparedAd.nativeContent` to a `NativeAdContent` carrying the creative assets (video URL, headline, body, CTA, sponsor name and icon, AdChoices icon, plus click and impression handlers). The SDK then plays the video through its own player pool and renders your ad overlay on top.

## Ad overlays

Native ads render UI through the `AdOverlay` protocol, configured with `FeedConfig.adOverlay` — the same overlay pattern as video, carousel, and survey cells. Your overlay receives the `NativeAdContent` and is responsible for the required policy elements (an "Ad" badge and a tappable AdChoices icon). See [item overlays](/docs/guides/customize/item-overlays) for the full protocol and lifecycle.
