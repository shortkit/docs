ShortKit ships five client SDKs — iOS, Android, Flutter, React Native, and Web. They share one setup model: install the package, create a ShortKit instance with your publishable key, then mount a feed. The package, language, and exact API differ per platform, so follow your platform's guide for the concrete steps.

## Set up the SDK

1. **Get your publishable key.** Create one in the portal — it looks like `pk_live_your_publishable_key`. Publishable keys are safe to ship in client apps; see [Authentication](/docs/api/authentication).
2. **Install the package** for your platform (table below).
3. **Initialize ShortKit** with your key and mount a feed. Each platform's guide has a copy-paste first feed.

Notes on specific rows:

- **Player and widgets on Android** — Android has no standalone player display surface or widget components today. The feed embed (`ShortKitFeedFragment`) is the only display surface.
- **Item overlays** — custom overlays for image carousels and video carousels. The Web SDK does not yet render carousel items in the feed, so it has no item-overlay surface.
- **Carousel control** — the programmatic carousel API (active index, next/previous, completion events). Flutter renders carousels and supports carousel overlays but does not expose the carousel controller.
- **Live viewing** — the in-feed live surface (live badge, viewer count, live-room behavior) ships on iOS and Web only.
- **Live broadcast** — broadcasting is server-side on every platform: you [create a live stream](/docs/api/live-streams/create-live-stream) through the REST API and push to its RTMP ingest URL from encoder software. No SDK exposes a client-side broadcast surface.
- **Downloads** — on-device video downloads with progress callbacks ship on iOS and React Native. The REST API also exposes [download URLs](/docs/api/content/get-download-url) for server-side use on any platform.
- **Surveys** — the built-in survey overlay surface ships on iOS and Android. Surveys themselves are created and managed through the [REST API](/docs/api/surveys/create-survey) regardless of platform.

## How the docs mark availability

### The "Not yet supported" tab convention

Feature pages describe each feature once, in platform-agnostic prose, with a tab per platform for code. Every page uses the same five tab labels in the same order: **iOS, Android, Flutter, React Native, Web**.

A platform tab contains one of exactly two things:

1. Real, shipped API for that platform.
2. A single line: *Not yet supported on \{platform\}.* — usually with a link back to this page.

A tab never shows speculative or analogized code for a platform that lacks the feature. If you see working code in a tab, it reflects the shipped SDK at the versions listed above.

Where a feature is exclusive to a subset of platforms (for example, live viewing on iOS and Web), the page says so once in a callout and tabs only the supported platforms.

### Version annotations

- **Since X** — the API was added in SDK version X. Integrations pinned below X won't have it.
- **Unreleased** — the API is merged but not yet published to the package registry. Don't build against it until the version that includes it ships.
