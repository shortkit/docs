<h3>Video carousel FeedInput now accepts playback IDs only (v0.3.0)</h3>
<p>The <code>videoCarousel</code> FeedInput variant no longer accepts fully-constructed <code>ContentItem[]</code> objects. The <code>videos</code> field on <code>VideoCarouselInput.item.videos</code> now takes <code>VideoCarouselVideoInput[]</code> — <code>{"{ playbackId, fallbackUrl? }"}</code> — and the SDK constructs <code>streamingUrl</code> and <code>thumbnailUrl</code> internally via its canonical <code>stream.media.shortkit.dev</code> / <code>image.media.shortkit.dev</code> domains. This matches how the <code>video</code> FeedInput variant already worked. Overlays are unaffected: <code>VideoCarouselOverlayProps.carouselItem</code> continues to provide the fully-hydrated <code>VideoCarouselItem</code> shape with <code>ContentItem[]</code> videos. Version matrix: <code>@shortkitsdk/react-native</code> 0.3.0, Swift SDK 0.2.0, Android SDK 0.3.0, Flutter SDK 0.3.0. See the <a href="/migration/v0.3.0-carousel-playback-ids">migration guide</a>.</p>
</div>

<h3>Fallback URLs</h3>
<p>Custom feeds now support optional fallback URLs on video items. When the primary HLS stream fails, the SDK automatically switches to a static MP4 fallback. Supports three activation modes: reactive (on player error), proactive (pre-detecting failure on the next video), and immediate (when the primary URL is missing). Available on iOS, Android, and React Native.</p>
