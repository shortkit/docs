The Feed API is the primary endpoint consumed by the ShortKit SDK. It returns a ranked list of video content tailored to the requesting user. The Configuration endpoints deliver a sparse config overlay to the SDK at runtime, letting you change player behavior, theming, and feed settings without shipping an app update.

## Endpoints

| Method | Endpoint | Auth | Description |
|--------|----------|------|-------------|
| `GET` | `/v1/feed` | Publishable key | Get a personalized video feed |
| `GET` | `/v1/config` | Publishable key | Retrieve the active config overlay |
| `PUT` | `/v1/config` | Secret key | Update the config overlay |

---

## Get feed

Retrieves a ranked feed of video content for a given user. The SDK calls this endpoint automatically, but you can also call it directly for server-side rendering or custom integrations.

```bash
GET /v1/feed
```

### Authentication

The Feed API uses a **publishable key** passed via the `X-API-Key` header. Publishable keys are safe to embed in client-side code and mobile apps.

```bash
X-API-Key: pk_live_your_publishable_key
```

### Query parameters

<ParamField query="userId" type="string">
  Identifier for the user requesting the feed. When provided, the ranking algorithm personalizes results based on the user's watch history and engagement signals. If omitted, the feed returns a default ranked order.
</ParamField>

### Example request

```bash
curl "https://api.shortkit.dev/v1/feed?userId=user_456" \
  -H "X-API-Key: pk_live_your_publishable_key"
```

### Feed item object

Each item in the `data` array contains:

<ResponseField name="id" type="string">
  Content identifier (e.g., `sk_content_abc123`).
</ResponseField>

<ResponseField name="title" type="string">
  Display title.
</ResponseField>

<ResponseField name="description" type="string | null">
  Optional description text.
</ResponseField>

<ResponseField name="duration" type="number">
  Video duration in seconds.
</ResponseField>

<ResponseField name="streamingUrl" type="string">
  HLS streaming URL. Pass this directly to your video player.
</ResponseField>

<ResponseField name="thumbnailUrl" type="string">
  Thumbnail image URL for the poster frame.
</ResponseField>

<ResponseField name="captionTracks" type="array">
  Available caption tracks. Each track has `language` (ISO 639-1 code), `label` (human-readable name),
  `source` (`"embedded"` for captions in the HLS stream, `"external"` for sidecar files),
  and `url` (`null` for embedded, a URL for external).
</ResponseField>

<ResponseField name="customMetadata" type="object">
  Arbitrary key-value pairs set when the content was created or updated.
  Use this to pass application-specific data (article URLs, category tags, deep links)
  through to your app's UI layer.
</ResponseField>

## Ad slots in the feed

When your organization has ads enabled via the `/v1/ad-config` endpoint, the feed response can include `ad_slot` items interleaved with `content` items. Each item in the `data` array carries a `type` field that acts as a discriminator:

- `"content"` -- A standard video content item, with the schema documented above.
- `"ad_slot"` -- An ad placement marker inserted by the server at configured intervals.

<Note>
  If you are using the ShortKit SDK, ad slots are handled automatically. The SDK loads and renders ads through the configured ad provider. This section is for customers calling the Feed API directly.
</Note>

### Ad slot object

When `type` is `"ad_slot"`, the item contains the following fields:

<ResponseField name="id" type="string">
  Server-assigned ad slot identifier (e.g., `ad_slot_001`).
</ResponseField>

<ResponseField name="position" type="integer">
  The position of this ad slot in the feed.
</ResponseField>

<ResponseField name="adTagUrl" type="string | null">
  VAST tag URL for the ad request. Used by ad providers such as Google IMA to fetch ad creatives.
</ResponseField>

<ResponseField name="adUnitId" type="string | null">
  Ad unit identifier. An alternative to `adTagUrl` for ad providers that use unit-based lookups.
</ResponseField>

<ResponseField name="targeting" type="object">
  Key-value pairs for ad targeting (e.g., `{"content_category": "sports", "org": "espn"}`). Passed to the ad provider to refine ad selection.
</ResponseField>

<ResponseField name="skipPolicy" type="string">
  Controls whether the user can skip the ad. One of:
  - `"non_skippable"` -- The ad cannot be skipped.
  - `"skippable_after"` -- The ad can be skipped after `skipDelaySeconds` seconds.
  - `"always_skippable"` -- The ad can be skipped immediately.
</ResponseField>

<ResponseField name="skipDelaySeconds" type="integer | null">
  Number of seconds the user must wait before skipping. Only applies when `skipPolicy` is `"skippable_after"`.
</ResponseField>

## Ranking

The feed ranking algorithm determines the order in which content is returned. It considers multiple signals to deliver the most relevant content to each user.

### Ranking signals

| Signal | Description |
|--------|-------------|
| **Recency** | Newer content is favored over older content. |
| **Editorial controls** | Pinned content appears at its designated position. Suppressed content, future-scheduled content, and expired content are excluded. |
| **Watch history** | Content the user has already watched is deprioritized to avoid repetition. |
| **Engagement** | Content with higher engagement rates (completions, likes, shares) is ranked higher. |
| **Personalization** | The algorithm learns from a user's viewing patterns to surface content aligned with their interests. |

### Content eligibility

Content must meet all of the following criteria to appear in the feed:

| Rule | Behavior |
|------|----------|
| **Status** | Only content with status `ready` is included. |
| **Suppression** | Content where `isSuppressed` is `true` is excluded. |
| **Publish window** | Content with a `publishAt` timestamp in the future is excluded. |
| **Expiration window** | Content with an `expiresAt` timestamp in the past is excluded. |

### Pinned content

Content with a `pinPosition` value is placed at its specified position in the feed, regardless of other ranking signals. Multiple pinned items are ordered by `pinPosition` ascending. All unpinned content follows, ordered by the ranking algorithm.

## SDK integration

The ShortKit SDK handles feed fetching and video playback automatically. You do not need to call the Feed API directly unless you are building a custom integration.

    let feedView = ShortKitFeedView()

## Feed error responses

| HTTP Status | Error code | Description |
|-------------|------------|-------------|
| 401 | `invalid_api_key` | The `X-API-Key` header is missing or the key is not valid. |
| 429 | `rate_limited` | Too many requests. Retry after the duration in the `Retry-After` header. |
| 500 | `internal_error` | An unexpected error occurred. Contact support if it persists. |

All error responses follow the standard envelope format:

```json
{
  "data": null,
  "meta": {
    "request_id": "req_err123"
  },
  "errors": [
    {
      "code": "invalid_api_key",
      "message": "The provided API key is not valid.",
      "status": 401
    }
  ]
}
```

## Remote configuration

The Configuration API delivers a sparse config overlay to the SDK at runtime. The overlay contains only the fields you have explicitly changed -- the SDK merges it on top of its compiled-in defaults. This means you can change player behavior, theming, and feed settings without shipping an app update.

### How the overlay works

The SDK ships with sensible defaults for every configurable property. When the SDK initializes, it calls `GET /v1/config` to fetch any overrides you have set. The response is **sparse** -- it only includes fields that differ from defaults. The SDK deep-merges the overlay on top of its defaults.

```
SDK defaults (compiled-in)
    + config overlay (from GET /v1/config)
    -----------------------------------------
    = effective configuration
```

For example, if the SDK default for `controls.scrubber` is `true` and you set it to `false` via the overlay, the overlay response contains only that field. Every other setting keeps its default value.

## Get config overlay

Retrieve the current config overlay for your organization. The SDK calls this endpoint automatically on initialization.

```bash
GET /v1/config
```

### Authentication

Include your publishable key in the `X-API-Key` header.

### Headers

| Header | Description |
|--------|-------------|
| `X-API-Key` | Your publishable key (`pk_live_...` or `pk_test_...`) |
| `X-User-Id` | Optional. User identifier for future per-user experiment assignments. |

### Example request

```bash
curl https://api.shortkit.dev/v1/config \
  -H "X-API-Key: pk_live_your_publishable_key"
```

### Example response

The response contains only the fields that differ from SDK defaults:

```json
{
  "data": {
    "controls": {
      "scrubber": false
    },
    "theme": {
      "primaryColor": "#FF5733"
    }
  },
  "meta": {
    "request_id": "req_abc123"
  }
}
```

If no overrides have been set, the response is an empty object:

```json
{
  "data": {},
  "meta": {
    "request_id": "req_def456"
  }
}
```

### Response fields

<ResponseField name="data" type="object">
  Sparse config overlay. Contains only fields that differ from SDK defaults. Any category or property not present should be left at its default value.
</ResponseField>

<ResponseField name="data.controls" type="object">
  Player control overrides.
  <Expandable title="properties">
    <ResponseField name="scrubber" type="boolean">
      Show the scrubber (seek bar) in the player. SDK default: `true`.
    </ResponseField>
    <ResponseField name="muteButton" type="boolean">
      Show the mute/unmute button. SDK default: `true`.
    </ResponseField>
    <ResponseField name="shareButton" type="boolean">
      Show the share button. SDK default: `true`.
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="data.playback" type="object">
  Playback behavior overrides.
  <Expandable title="properties">
    <ResponseField name="autoplay" type="boolean">
      Auto-start playback when content is visible. SDK default: `true`.
    </ResponseField>
    <ResponseField name="muted" type="boolean">
      Start playback muted. SDK default: `true`.
    </ResponseField>
    <ResponseField name="loop" type="boolean">
      Loop video on completion. SDK default: `true`.
    </ResponseField>
    <ResponseField name="preloadCount" type="integer">
      Number of videos to preload ahead (1-10). SDK default: `3`.
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="data.captions" type="object">
  Caption display overrides.
  <Expandable title="properties">
    <ResponseField name="defaultOn" type="boolean">
      Show captions by default. SDK default: `false`.
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="data.theme" type="object">
  UI theming overrides.
  <Expandable title="properties">
    <ResponseField name="primaryColor" type="string">
      Primary accent color (hex). SDK default: `"#FFFFFF"`.
    </ResponseField>
    <ResponseField name="backgroundColor" type="string">
      Background color (hex). SDK default: `"#000000"`.
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="data.feed" type="object">
  Feed behavior overrides.
  <Expandable title="properties">
    <ResponseField name="pageSize" type="integer">
      Items per feed request (10-50). SDK default: `10`.
    </ResponseField>
    <ResponseField name="refreshInterval" type="integer">
      Auto-refresh interval in seconds. SDK default: `300`.
    </ResponseField>
  </Expandable>
</ResponseField>

### Merge example

Given these SDK defaults:

```json
{
  "controls": { "scrubber": true, "muteButton": true, "shareButton": true },
  "playback": { "autoplay": true, "muted": true, "loop": true, "preloadCount": 3 },
  "captions": { "defaultOn": false },
  "theme": { "primaryColor": "#FFFFFF", "backgroundColor": "#000000" },
  "feed": { "pageSize": 10, "refreshInterval": 300 }
}
```

And this overlay from `GET /v1/config`:

```json
{
  "controls": { "scrubber": false },
  "theme": { "primaryColor": "#FF5733" }
}
```

The SDK resolves the effective configuration as:

```json
{
  "controls": { "scrubber": false, "muteButton": true, "shareButton": true },
  "playback": { "autoplay": true, "muted": true, "loop": true, "preloadCount": 3 },
  "captions": { "defaultOn": false },
  "theme": { "primaryColor": "#FF5733", "backgroundColor": "#000000" },
  "feed": { "pageSize": 10, "refreshInterval": 300 }
}
```

Only `controls.scrubber` and `theme.primaryColor` changed. Everything else kept its default.

## Update config overlay

Set or update the config overlay for your organization. Only include the fields you want to override -- omitted fields are removed from the overlay (reverting to SDK defaults).

```bash
PUT /v1/config
```

### Authentication

Requires your **secret key** as a Bearer token.

### Example request

```bash
curl -X PUT https://api.shortkit.dev/v1/config \
  -H "Authorization: Bearer sk_live_your_secret_key" \
  -H "Content-Type: application/json" \
  -d '{
    "controls": {
      "scrubber": false,
      "shareButton": false
    },
    "playback": {
      "preloadCount": 5
    },
    "captions": {
      "defaultOn": true
    },
    "theme": {
      "primaryColor": "#FF5733"
    }
  }'
```

### Response

Returns the full overlay as saved:

```json
{
  "data": {
    "controls": {
      "scrubber": false,
      "shareButton": false
    },
    "playback": {
      "preloadCount": 5
    },
    "captions": {
      "defaultOn": true
    },
    "theme": {
      "primaryColor": "#FF5733"
    }
  },
  "meta": {
    "request_id": "req_xyz789"
  }
}
```

### Reset to defaults

To reset all overrides and revert entirely to SDK defaults, send an empty object:

```bash
curl -X PUT https://api.shortkit.dev/v1/config \
  -H "Authorization: Bearer sk_live_your_secret_key" \
  -H "Content-Type: application/json" \
  -d '{}'
```

## SDK sync

The SDK fetches the config overlay automatically on initialization and refreshes it periodically (default: every 5 minutes). Configuration changes take effect on the next SDK refresh without requiring an app update.

## Best practices

<AccordionGroup>
  <Accordion title="Keep overlays small">
    Only override what you need. A smaller overlay means faster config resolution and less risk of unintended side effects.
  </Accordion>
  <Accordion title="Test in staging first">
    Use your `pk_test_...` key to verify configuration changes before applying them in production.
  </Accordion>
  <Accordion title="Use A/B testing for significant changes">
    For changes that may affect engagement (hiding controls, changing preload behavior), use experiments to roll out to a subset of users first.
  </Accordion>
</AccordionGroup>
